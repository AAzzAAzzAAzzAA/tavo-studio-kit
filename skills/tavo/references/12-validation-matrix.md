## Evidence Levels

- `claimed`: stated in current official docs.
- `needs-live-verify`: official docs or historical-derived guidance suggest the test target, but the specific behavior has not reached MCP schema, dry-run, UI, or roundtrip proof.
- `schema-seen`: current MCP exposes a matching schema or tool.
- `dry-run-pass`: current MCP dry-run accepted the object.
- `live-verified`: a scoped behavior was executed on the current real app and backed by direct readback or retained runtime artifacts; it is not automatically semantic, visual, or persistence proof.
- `live-verified-regression`: a scoped current behavior contradicted the declared or previously observed effect with direct readback; preserve both the positive and negative evidence.
- `ui-pass`: Android app shows the expected result.
- `roundtrip-pass`: export/import or create/read cycle preserved the intended data.
- `semantic-pass`: a real model exchange with isolated expected/forbidden markers, stable persistent message IDs, and required state readback passed.
- `semantic-pass-observation`: a real model saw an isolated marker under a configured field/position/depth, but the evidence does not prove exact prompt order, wire role, or causation.
- `semantic-mixed`: current retained semantic runs include both a positive observation and a reproducible regression; do not publish an unconditional capability claim.
- `not-applicable`: the case targets another platform or an unavailable environment; it is not a pass, failure, or blocked runtime claim.

## Recording Rule

When a row is tested, record the date, app version, official doc snapshot path, MCP dump path, Android evidence path, retained object/file names, and any restore action. Keep raw secrets out of evidence files.

The current project preference is to retain real-phone validation files and disposable test objects as evidence. Do not delete them by default. Only run cleanup when the matrix row is explicitly about cleanup/restore or the user asks for cleanup.

## Status And Evidence Separation

`Status` tracks workflow state: `pending`, `planned`, `running`, `blocked`, `passed`, or `failed`.

`Evidence level` tracks the proof axis, not a single promotion ladder: `needs-live-verify`, `claimed`, `schema-seen`, `dry-run-pass`, `live-verified`, `live-verified-regression`, `ui-pass`, `roundtrip-pass`, `semantic-pass-observation`, `semantic-pass`, or `semantic-mixed`.

Do not upgrade `Status` to `passed` unless the row also records the artifact path, app version, retention/restore result, and evidence level.
