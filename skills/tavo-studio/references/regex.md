# Tavo 正则总结

## 1. 这是什么

Tavo 的正则功能用于自动识别、清洗、替换和重写文本。它本质上是一套可配置的文本处理流水线，可以作用在用户消息、角色消息、推理内容和世界书注入内容上。

如果后续要做：

- 输入简写展开
- 输出格式统一
- 去掉模型多余前缀
- 拼接状态栏
- 将某些文本转成变量宏

那正则会非常关键。

边界先写死：正则是文本替换流水线，不是 JS 执行器，也不是状态存储系统。它可以把文本替换成宏或 HTML，但宏是否会在后续 prompt/send 链路执行、HTML/JS 是否会在高级前端渲染里运行，要看执行时机、placement、AR 设置和沙盒限制。

## 2. 主要用途

文档强调它主要用于三件事：

- 识别特定文本模式
- 修改或替换文本内容
- 清理多余字符和格式

常见应用场景：

- 去掉 `[系统提示]` 之类的冗余前缀
- 把用户输入的简写转成完整命令
- 清洗模型输出中的多余格式
- 为输出追加统一状态栏
- 对世界书注入内容再做一层格式加工

## 3. 配置项逐条总结

## 3.1 名称

- 只是规则名
- 用于区分和管理多个规则
- 最好按用途命名，比如“清理系统提示”“状态栏拼接”“输入宏展开”

## 3.2 正则表达式（Find Regex）

- 定义匹配模式
- 用来告诉系统“哪些文本应该被命中”
- 文档示例是 `^\\[系统提示\\]：(.+)$`

这个示例的含义是：

- `^` 匹配行首
- `\\[系统提示\\]` 匹配字面文本 `[系统提示]`
- `(.+)` 捕获后面的正文
- `$` 匹配行尾

## 3.3 替换内容（Replace With）

- 定义命中后替换成什么
- 可以使用捕获组，如 `$1`、`$2`
- 也可以使用 `{{match}}` 表示完整匹配内容

文档示例中：

- 原文：`[系统提示]：任务开始`
- 正则：`^\\[系统提示\\]：(.+)$`
- 替换：`$1`
- 最终显示：`任务开始`

## 3.4 替换前修剪（Trim Out）

- 在正式替换前，先移除一些固定字符串
- 适合清理杂质前缀
- 文档示例是去掉“玩家说：”

## 3.5 作用范围

文档列出的作用范围有：

- 用户消息
- 角色消息
- 思维链
- 世界书

也就是说，正则不只是处理最终可见文本，也可以处理模型内部使用的文本来源。

## 3.6 执行时机

文档列出的执行时机有：

- 显示时
- 发送时
- 发送和显示时
- 接收时
- 接收和改写时

这几个时机很重要，因为它决定了修改是：

- 只影响界面显示
- 只影响送进模型的内容
- 影响保存下来的实际消息

## 3.7 替换参数

文档列出三种替换参数：

- 不替换
- 原文替换
- 转义替换

这部分本质上是控制“替换后的文本如何处理”，尤其和特殊字符的保留、转义有关。

## 4. 官方给出的综合示例

官方示例是清理用户消息中的系统前缀：

- 目标：去掉 `[系统提示]：`
- 正则：`^\\[系统提示\\]：(.+)$`
- 替换内容：`$1`
- 作用范围：用户消息
- 执行时机：显示时
- 替换参数：原文替换

结果是：

- 输入显示前先被加工
- 用户看到的只有“任务开始”

## 5. 对后续实操最有价值的用法

### 5.1 清理输出噪音

适合去掉：

- 多余前缀
- 模型重复的角色名
- 不想显示给用户看的内部标记

### 5.2 输入简写展开

这点在宏文档也举了例子，比如：

- `-hp5` 展开成多个 `{{decvar::hp}}`
- `+gold20` 展开成 `{{addvar::gold::20}}`

这类玩法非常适合把正则当作“命令转换层”。注意：只有替换结果继续进入会执行宏的链路时，变量宏才会真正改状态；如果只是显示层替换，通常只影响用户看到的文本。

### 5.3 输出状态栏与格式统一

可以让角色输出自动附带：

- HP
- MP
- 回合数
- 地点
- 情绪状态

前提是先通过宏或变量系统维护状态。

不要用正则本身当状态机。正则可以拼接或显示状态栏，但状态的真源应来自宏变量或 JS API 变量。

### 5.4 世界书和正则联动

世界书负责按条件注入设定

正则负责把注入后或显示前的文本再整理成更适合阅读或建模的形式。

## 6. 和 JavaScript API 中 regex 数据结构的对应关系

如果后续要用脚本操作正则组，需要知道用户界面里的概念与 API 字段基本对应：

- 名称 -> `name`
- 查找正则 -> `findRegex`
- 替换内容 -> `replaceString`
- 修剪文本 -> `trimStrings`
- 作用范围 -> `placements`
- 执行时机 -> `timing`
- 替换参数 -> `substitution`
- 是否启用 -> `enabled`

API 里还额外支持：

- `minDepth`
- `maxDepth`

也就是按消息深度限定规则生效范围。

## 6.1 ST→Tavo 正则字段精确转换规则（基于源码）

以下映射来自 sandbox.js 中的 `w()` 转换函数，在调用 `tavo.regex.create()` / `tavo.regex.update()` 时自动发生（经 `c()` → `w()` 链路）。注意：`tavo.regex.import()` **不走** `w()` 转换，数据直接传给 Flutter 层。

### 字段名映射

| ST/旧字段 | Tavo 字段 | 说明 |
|-----------|----------|------|
| `scriptName` | `name` | 脚本名→规则名，通过 create/update 时自动转换 |
| `placement` (整数数组) | `placements` (字符串数组) | 见下方映射表 |
| `disabled` | `enabled` | **取反**：`disabled=true` → `enabled=false` |
| `markdownOnly` + `promptOnly` + `runOnEdit` | `timing` | 三个旧字段合并为一个 |
| `substituteRegex` (整数) | `substitution` (字符串) | 见下方映射表 |

### placements 映射表

| ST 整数值 | Tavo 字符串值 | 含义 |
|-----------|-------------|------|
| `1` | `"user"` | 用户输入 |
| `2` | `"char"` | AI 输出（角色消息） |
| `5` | `"lorebook"` | 世界书内容 |
| `6` | `"reasoning"` | 推理/思维链 |

### timing 映射表（三旧字段→一字段）

| markdownOnly | promptOnly | runOnEdit | 结果 timing |
|-------------|-----------|-----------|------------|
| `true` | `true` | — | `"sendAndDisplay"` |
| `true` | `false` | — | `"display"` |
| `false` | `true` | — | `"send"` |
| — | — | `true` | `"editAndReceive"` |
| 全 `false` | — | — | `"receive"` |

### substitution 映射表

| ST 整数值 | Tavo 字符串值 | 含义 |
|-----------|-------------|------|
| `0` | `"none"` | 不替换 |
| `1` | `"raw"` | 原文替换 |
| `2` | `"escaped"` | 转义替换 |

### 关键警告

- ST 旧格式的整数 `placement` 与 Tavo 新格式的字符串 `placements` 之间的对应关系见上表。两套格式各自用在不同的 API 入口：
- **两条路要分清**：
  - `tavo.regex.import()` — 吃 **ST 旧格式**：`placement`（**单数**）用**整数数组**（`[1, 2]`），`disabled` 用**布尔值**，旧 timing 三字段 `markdownOnly` / `promptOnly` / `runOnEdit`（sandbox.js 源码：`callParentWithResult("regex_import", [e])`，不经 `w()` 转换直接发送）
  - `tavo.regex.create()` / `tavo.regex.update()` — 用 **Tavo 新格式**：`placements`（**复数**）用**字符串数组**（`["char", "user"]`），`enabled` 用**布尔值**（语义与 `disabled` 相反），`timing` 用**字符串**（sandbox.js 源码：经 `c()` → `w()` 链路转换后发送）
- 生成正则 JSON 时确定目标方法：给 `import()` 用 ST 格式，给 `create/update()` 用 Tavo 格式
- 生成正则 JSON 时，**Tavo 格式必须使用 `enabled`**（不是 `disabled`），值为布尔类型
- 生成正则 JSON 时，**Tavo 格式必须使用 `timing`** 字符串值，不能使用旧的 `markdownOnly`/`promptOnly`/`runOnEdit` 三字段

## 7. 使用时的注意事项

- 正则配置错了会直接影响输入、输出或持久化消息
- “显示时”通常最安全，因为它不一定改动持久数据
- “接收时”或“接收和改写时”更有侵入性，要谨慎
- 复杂玩法里，建议先从单条规则开始调试
- 如果要做变量驱动玩法，优先把“状态变化”交给宏，把“用户简写/文本清洗”交给正则
- 不要默认认为正则替换出的 `<script>`、`onclick`、`onload` 会执行；高级前端渲染会经过 Markdown、DOMPurify、iframe 沙盒和 `javaScriptExecutionMode` 判断
- 要做可靠交互或写状态，优先用 `tavo.*` JS API；正则只负责把用户输入或模型输出整理成更适合处理的文本

## 8. 适合后续开发/配置时记住的点

- 正则是 Tavo 的文本中间层
- 它非常适合做“输入命令化”和“输出格式化”
- 与宏联动时可玩性很高
- 与世界书联动时可以做条件触发文本重构
